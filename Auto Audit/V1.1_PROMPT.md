## 1. Authentication (Supabase)

Add Supabase as the auth provider. Use `@supabase/supabase-js` and `@supabase/ssr` for Next.js App Router compatibility.

### Pages to add

- `app/(auth)/signup/page.tsx` — email + password + confirm password
- `app/(auth)/login/page.tsx` — email + password, "Forgot password?" link
- `app/(auth)/forgot-password/page.tsx` — email field, sends reset link
- `app/(auth)/reset-password/page.tsx` — new password + confirm, accepts the recovery token from the URL

All four pages use a shared minimal auth layout (`app/(auth)/layout.tsx`) with the Auto Audit logo, no app sidebar, and a clean centered card. Match the polish of the marketing landing page.

### Behavior

- After successful signup, send the user to the dashboard with their fresh empty account
- After successful login, send to dashboard
- Auth state lives in a `lib/AuthContext.tsx` provider that wraps `BudgetProvider`
- Session must persist across reloads (Supabase handles this via cookies — use `@supabase/ssr` with the SSR-safe client)
- Redirect rules:
  - `(app)` routes require auth OR demo mode (see section 2). Logged-out users hitting `/dashboard` directly get sent to `/login`
  - `(auth)` routes are unreachable while logged in — redirect to `/dashboard`
- Friendly inline form errors. Use the existing `Alert` component for top-of-form errors

### Storage adapter

The existing `StorageAdapter` interface in `lib/storage.ts` should now have **two implementations**:

1. `LocalStorageAdapter` (existing) — used in demo mode
2. `SupabaseStorageAdapter` (new) — used for authenticated users; reads/writes the user's data from Supabase tables

`BudgetContext` should pick which adapter to instantiate based on auth state (logged in → Supabase, demo mode → localStorage). The provider should swap cleanly without losing the same calling API.

### Supabase schema

Generate a `supabase/schema.sql` file with these tables, all with `user_id uuid references auth.users` and Row Level Security policies that scope every row to the owning user:

- `categories` (id, user_id, name, color, sort_order, is_default, created_at)
- `transactions` (id, user_id, amount, merchant, occurred_on, category_id, note, created_at)
- `monthly_budgets` (id, user_id, month_key, total, per_category jsonb, created_at, updated_at)
- `savings_goals` (id, user_id, name, target_amount, saved_amount, kind, created_at, updated_at)
- `merchant_memory` (id, user_id, normalized_merchant, category_id, last_seen_at)
- `user_settings` (user_id pk, advanced_mode bool, theme text default 'system', created_at, updated_at)

Include indexes on `(user_id, occurred_on)` for transactions and `(user_id, month_key)` for monthly budgets.

### Env vars

Add to `.env.local.example` (and a note in README):
```
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
```

## 2. Demo mode ("Try without signup")

The marketing landing page CTAs currently link straight into the app. Update them so the landing page now offers two clear paths:

- **Primary CTA:** "Get started" → `/signup`
- **Secondary CTA:** "Try the demo" → enters demo mode (no signup) and goes to `/dashboard`

Demo mode behavior:
- Loads Alex Carter's 12 months of data into the in-memory state
- Uses `LocalStorageAdapter` so edits persist in the browser, scoped to a `demo` namespace key (so they never collide with a real user's local cache)
- Shows a thin persistent banner at the top of the app: "You're in demo mode. Sign up to save your data." with a "Sign up" button
- All in-app features work — but signing out drops them back to the landing page, and no data is ever sent to Supabase

The existing "Reset demo" affordance should remain available in demo mode.

## 4. Dark mode

Add a light/dark theme system.

- Default to **system preference** on first visit (`prefers-color-scheme`)
- Manual toggle in the top header (sun/moon icon, single click flips it). The user's choice is saved (localStorage for demo mode, `user_settings.theme` for authenticated users) and overrides system preference once set
- Three states stored: `light`, `dark`, `system` — but the toggle only flips between `light` and `dark` after first manual interaction (a small dropdown or right-click menu can offer "Use system" if you want; otherwise just light/dark is fine)

### Implementation

- Use Tailwind's `darkMode: 'class'` strategy in `tailwind.config.ts`
- Add a `ThemeProvider` in `lib/ThemeContext.tsx` that toggles a `dark` class on `<html>`
- Extend the existing brand/warn/danger color scales with dark-mode equivalents
- All components must support both themes — audit each one. Cards, inputs, charts, progress bars, alerts, the printable report (print should always be light regardless of theme)

### Color guidance for dark mode

- Background: deep neutral (~#0B0D10 / Tailwind `neutral-950`-ish), not pure black
- Surface (cards): one step lighter (~#15181D)
- Text primary: near-white, not pure white
- Brand green: keep the same vivid green, slightly desaturated if it vibrates against dark surfaces
- Yellow/red alerts: bump saturation slightly so they read against dark surfaces
- Recharts: pass theme-aware colors via CSS variables or a `useTheme` hook so chart axes/grid lines flip too

## 5. Polish pass — empty states & transitions

### Empty states

Every surface where data could be missing needs a real designed empty state, not just blank space. Use a consistent pattern: small icon, headline, one-line description, and a primary CTA where it makes sense.

Audit and improve:
- Dashboard with no transactions for the selected month (already exists — make it warmer and visually polished)
- Transaction History with no transactions
- Transaction History after a search/filter returns zero results (different from "no transactions at all")
- Monthly Summary with no spending
- Savings Goals with no goals (already exists — refine)
- Printable Report when nothing happened that month
- New-user dashboard (covered in section 3)

### Transitions & micro-interactions

Add subtle motion. Nothing flashy — these should feel like the app is responding, not putting on a show.

- **Page transitions:** soft fade + 4-8px upward slide on route change inside `(app)` (use Framer Motion or CSS-based with `next/dynamic` patterns; Framer Motion is fine)
- **Card mounts:** stagger fade-in on dashboard cards (~30ms between siblings, total ~200ms)
- **Progress bars:** animate width on month change (300ms ease-out)
- **Modals/dialogs:** if any (transaction edit, delete confirm), scale-fade in
- **Toast notifications:** add a global toast system. Slide in from bottom-right, auto-dismiss 3s. Use for: expense added, expense deleted, budget saved, savings contribution made, fuzzy-match confirmation, sign-in / sign-out, password reset email sent, errors. Wire toasts into the existing context actions
- **Number changes:** when remaining budget or % used change, animate the digit (e.g. CountUp effect or simple easing). Optional but nice on the stat tiles

### Loading states

- App-level: a top thin progress bar during route transitions
- Auth forms: button shows a spinner + disables on submit
- Supabase data fetch: skeleton loaders on the dashboard cards while initial data loads (avoid layout shift)

### Accessibility checks while you're polishing

- All toggles, the dark mode switch, and the toast dismiss control need keyboard focus rings and `aria-label`s
- Color contrast in both themes meets WCAG AA for body text
- `prefers-reduced-motion` disables all transitions and animations

## 8. Acceptance criteria for v1.1

1. A new visitor lands on `/`, sees the polished landing page, and can choose **Try the demo** (loads Alex Carter, no auth) or **Sign up** (creates real Supabase account)

3. Login persists across browser reloads. Logout returns to the landing page
4. Forgot/reset password flow works end to end via Supabase email
5. The dark/light toggle works on every page including charts, the printable report renders correctly in both themes (or always light when printed), and the user's theme choice persists
6. Every empty state has a designed treatment (icon + headline + description + CTA where applicable)
7. Page changes, card mounts, and progress bars animate subtly. Toast notifications fire for all the listed actions. `prefers-reduced-motion` cleanly disables animations
8. `npx tsc --noEmit` passes with zero errors
9. `npx next build` completes successfully
10. The demo mode banner is visible and dismisses correctly when the user signs up
